1. Project Architecture

The scheduler is divided into three separate processes for horizontal scalability, decoupling long-running compute jobs from HTTP request/response handling, and single-leader operations:

  • API Server (Stateless): Serves authenticated REST requests, processes incoming payloads, and manages the WebSocket real-time gateway. Built using Node.js, Express, and Socket.IO.
  • Worker Daemon: Independent fleet processes that poll database queues using atomic transactions, execute tasks concurrently, and periodically dispatch health heartbeats.
  • Scheduler Process: A single-leader coordinator (elected using a Redis lease lock) that performs time-based triggers: promoting scheduled jobs, generating cron definition intervals, and reclaiming dead leases from crashed workers.

Reliability & Concurrency Model

Concurrency limits and atomic claims are handled entirely at the database layer using Postgres transaction locking instead of unsafe memory flags. When claiming a job, the query isolates rows in a transaction using FOR UPDATE SKIP LOCKED, allowing multiple workers to safely poll the database in parallel without duplicate claim errors.

2. Relational Database Design

Below is the summary of the main entities used in the application. Indexes are built using partial constraints on status fields to keep index tables highly optimized:

queues UUID project_id (CASCADE) Composite uniqueness on (project_id, name)

4. Testing & Concurrency Proofs

Automated tests are located in backend/tests and run against a live local Postgres database instance:

  • concurrency.test.ts: Spawns 25 concurrent worker processes polling 200 jobs from a single queue to assert that no duplicate execution occurs.
  • backoff.test.ts: Validates linear, fixed, and exponential delay math with and without jitter.
  • api.test.ts: Integration test verifying rate limiting, project-scoping, and routing rules.

5. Critical Design Decisions

  1. Raw SQL queries instead of ORMs: Adopted raw SQL via pg. Standard ORMs make it difficult to structure complex CTE queries containing FOR UPDATE SKIP LOCKED alongside custom concurrency checks.
  2. Direct Worker connection to database: Workers talk directly to the database instead of HTTP routes to avoid authentication overhead and packet latency on fast poll ticks.
  3. Dead Letter Queue separation: Implemented DLQ records as their own table (`dead_letter_entries`) holding payload snapshots. This maintains query performance on the dashboard and ensures records survive future job log retention purges.
